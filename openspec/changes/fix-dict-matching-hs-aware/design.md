## Context

Currently, the data cleaning pipeline uses `DictionaryMatcher` to classify products based on an Aho-Corasick automaton. The matching logic relies purely on substring matches within the product description against the `TONG_THE.csv` dictionary. This approach has proven highly inaccurate (Dictionary only correct 0% of the time without AI) because generic keywords generated by the dictionary generator (like "mặt trời hiệu" or "gp mosquito") cause severe false positive matches across unrelated product categories. The fundamental issue is that the matcher is completely unaware of the product's `Mã HS` (HS Code) during the matching phase, whereas the source dictionary was built using HS Code as a primary grouping mechanism.

## Goals / Non-Goals

**Goals:**
- Update `DictionaryMatcher.predict` to accept and utilize `Mã HS` for contextual filtering.
- Modify the Aho-Corasick payload to store the `Mã HS` from the dictionary entry.
- Implement logic to drastically penalize or discard keyword matches if the 4-digit HS prefix of the input product does not match the 4-digit HS prefix of the dictionary entry.
- Update downstream callers (e.g., `data_cleaner.py` and `worker.py`) to pass the HS code to the matcher.
- Allow fallback to standard matching if the input product has no HS code.

**Non-Goals:**
- Completely overhauling the TF-IDF dictionary generator logic (this is a downstream fix).
- Changing the PhoBERT AI model logic or its calibration (to be handled separately).

## Decisions

**1. Include HS Code in the Aho-Corasick Payload**
- **Rationale**: To compare the input product's HS code against the matched keyword's HS code in `O(1)` time upon a match, the `DictionaryMatcher._load_dict` method must store the dictionary entry's `Mã HS` directly in the payload added to the Aho-Corasick automaton.

**2. Modify `predict(self, text, hs_code=None)` Signature**
- **Rationale**: `DictionaryMatcher.predict` needs the HS code to perform the filtering. The signature must be updated to accept `hs_code`.

**3. Prefix-Based HS Code Matching (4-Digit)**
- **Rationale**: HS codes can be 8, 6, or 4 digits. Comparing exact 8-digit matches is too strict. Comparing the 4-digit prefix (e.g., '9405' vs '9405') provides a good balance of contextual filtering while allowing reasonable generalization within the same product family. If the 4-digit prefix of the input does not match the 4-digit prefix of the dictionary entry, the match score will be heavily penalized (e.g., divided by 10) or discarded. Discarding is preferred for absolute strictness.

**4. Update Callers**
- **Rationale**: In `backend/core/worker.py` and `backend/core/data_cleaner.py`, the `apply(matcher.predict)` logic needs to be updated. Since `apply` on a Series only passes the text, it needs to be updated to apply on the DataFrame row to pass both text and HS code.

## Risks / Trade-offs

- **[Risk] Reduced Match Rate**: Strict HS filtering might reduce the overall number of dictionary matches, sending more products to the AI model.
  - **Mitigation**: This is an acceptable trade-off because current matches are mostly false positives (precision 17.45%). Sending to AI is better than an incorrect dictionary classification.
- **[Risk] Missing or Malformed HS Codes**: Input data might have NaN, missing, or malformed HS codes.
  - **Mitigation**: Add a fallback check. If `hs_code` is missing, NaN, or shorter than 4 digits, bypass the HS filter and rely on the existing substring matching.
